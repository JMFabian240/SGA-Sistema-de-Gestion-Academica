# RF-60: Registrar un pago

| Campo | Descripción |
| :---: | ----- |
| **Autor** | Diana Jareth Ritt Martinez (Actualizado por Antigravity) |
| **Nombre** | RF-60: Registrar un pago |
| **Actor** | Administrador o Gestor Administrativo |
| **Objetivo** | Permitir al personal registrar un pago realizado por un padre/tutor (ya sea el monto exacto, un abono parcial o un monto mayor con saldo a favor), seleccionando conceptos del catálogo estándar (Inscripción, Arancel, Material, Uniforme, Colegiatura, Libros) o ingresando conceptos personalizados ("Otros"), para actualizar el saldo pendiente y el historial financiero. |
| **Flujo Principal** | 1. El usuario accede a la pantalla de Caja en el Panel Principal.<br>2. Busca al tutor o alumno en el buscador.<br>3. El sistema muestra la ficha del tutor y el listado de adeudos pendientes (`CalendarioPago`) de sus alumnos asociados.<br>4. El usuario selecciona uno o varios conceptos a cobrar.<br>5. El sistema calcula automáticamente la suma total de los conceptos seleccionados y la muestra.<br>6. El usuario ingresa el monto total recibido, la fecha y el método de pago.<br>7. El sistema valida que el monto total sea mayor que cero.<br>8. El usuario confirma el registro de la operación.<br>9. El sistema procesa la transacción de cobro, descontando los montos aplicados de los adeudos correspondientes.<br>10. El sistema genera el comprobante digital de pago (ticket) y registra la transacción en la bitácora.<br>11. El sistema actualiza los saldos y KPIs del Panel Principal en tiempo real. |
| **Flujo Alterno** | **Flujo alterno A — El monto recibido es menor al total seleccionado (Abono)**<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario ingresa un monto total menor que la suma de los conceptos seleccionados.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema valida que el monto sea mayor que cero.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema aplica el pago de forma prorrateada o secuencial a los adeudos seleccionados según el orden de vencimiento.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El sistema descuenta el importe de los saldos pendientes, manteniendo los conceptos parcialmente cubiertos en estado **PENDIENTE** y actualizando el `saldoPendiente`.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El flujo continúa en el paso 10 del flujo principal.<br><br>**Flujo alterno B — El monto recibido cubre exactamente el total**<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario ingresa el monto exacto.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema actualiza el saldo de los conceptos seleccionados a cero y cambia su estado de cobro a **PAGADO**.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El flujo continúa en el paso 10 del flujo principal.<br><br>**Flujo alterno C — El monto recibido supera el total de adeudos (Saldo a Favor)**<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario ingresa un monto total mayor que la suma de los conceptos seleccionados.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema calcula la diferencia como excedente (exceso de pago).<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema liquida al 100% los adeudos seleccionados (cambiando su estado a **PAGADO** y saldo a cero).<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El sistema abona la diferencia al `saldoAFavor` en la cuenta del Tutor y registra un movimiento de ingreso en el historial de saldos.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El flujo continúa en el paso 10 del flujo principal.<br><br>**Flujo alterno D — Pago de colegiatura fuera de plazo de gracia**<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El sistema detecta que la fecha actual o de pago supera el límite del periodo de gracia (5 días hábiles después del vencimiento).<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema añade el recargo de $400.00 MXN al adeudo del alumno de forma automática.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema recalcula el total a cobrar en la interfaz.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El flujo continúa en el paso 5 del flujo principal.<br><br>**Flujo alterno E — No se encuentran resultados en el buscador**<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario introduce criterios de búsqueda.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema no encuentra ningún tutor o alumno que coincida.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema muestra un mensaje indicando que no hay coincidencias.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El usuario puede limpiar el campo y reintentar.<br><br>**Flujo alterno F — El usuario cancela el cobro**<br>&nbsp;&nbsp;&nbsp;&nbsp;1. En cualquier momento antes de confirmar, el usuario pulsa en "Cancelar".<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema cancela la operación y no guarda cambios en la base de datos. |
| **Precondiciones** | 1. El usuario (Administrador/Gestor) debe estar autenticado en el sistema.<br>2. El alumno y su tutor deben estar vinculados en la base de datos.<br>3. Deben existir conceptos de cobro o adeudos generados previamente (`CalendarioPago`).<br>4. Las tarifas aplicables al ciclo escolar activo deben estar configuradas. |
| **Postcondiciones** | • Se almacena el registro del Pago y las Aplicaciones de Pago asociadas de forma atómica.<br>• El saldo pendiente de los adeudos cobrados se actualiza (pasa a cero si se liquida).<br>• Si hay excedente, se incrementa el saldo a favor en la ficha del tutor.<br>• El estado del adeudo pasa a **PAGADO** si se cubre por completo, o permanece **PENDIENTE** si es parcial.<br>• El Panel Principal actualiza la sumatoria de ingresos del día y deudores de manera inmediata. |
| **Reglas de negocio involucradas** | • Los conceptos permitidos son Inscripción, Arancel, Material, Uniforme, Colegiatura, Libros, y otros personalizados ("Otros").<br>• Se permiten pagos parciales (abonos) que disminuyen el saldo pendiente sin cambiar el estado a PAGADO.<br>• Los pagos excedentes generan saldo a favor en la cuenta del tutor de forma automática.<br>• No se permiten registrar pagos duplicados sobre conceptos con estado PAGADO.<br>• Se aplica recargo por mora de $400.00 MXN tras vencer el periodo de gracia (5 días hábiles).<br>• La regla de descuento por hermanos (30%) y promociones de inscripción (matriz por grado/nivel) son excluyentes (no se duplican sobre el mismo alumno). |
